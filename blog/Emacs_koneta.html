日本語文字コードの設定

  1. まず、文字コードの設定で最初に行うのは、言語環境を日本語(Japanese)に設定することです。
    「何故、文字コードの設定で言語環境を設定するか」というと、 言語環境を設定すれば最低限の文字コードの設定が行われるからです。
    言語環境の設定は init.el などの設定ファイルに set-language-environment で指定します。
    言語環境を日本語に設定すれば、環境に合わせた日本語文字コードが設定されます。
    Windows では Shift-JIS 、Unix 系では EUC-JP などユーザーが環境として選んでいる文字コードです。
    より細かな設定をするためにはこの後に文字コードの設定を追加していくことになります。

  2. 文字コードのシンボル
    文字コードはシンボルを使って指定します。

    シンボル 文字コード
    shiftjis Shift-JIS
    cp932 コードページ932 (Windows では正確にはこちら)
    euc-jp EUC-JP
    utf-8 UTF-8 (BOM なし)
    utf-8-with-signature BOM 付き UTF-8

    また、改行コードは指定がなければシステムに合わせたものになります。
    これを変えたい場合は shiftjis-unix のように以下のキーワードと合わせたシンボルを指定します。

    キーワード 説明
    dos CR + NL
    mac CR
    unix NL

  3. 対象別の文字コード指定
    デフォルトの文字コードを設定する場合には prefer-coding-system を使います。

    (prefer-coding-system 'utf-8)

    同じような関数に set-default-coding-systems というのもあります。
    ただし、ヘルプによれば prefer-coding-system の方は改行コードとのセットのシンボルも使えるようですので、 こちらを覚えておいた方がいいでしょう。

    この設定によって指定される文字コードは以下の項目です。

    • ファイルを新規作成した場合のデフォルト
    • サブプロセスでの IO
    • 他の項目が指定されていない場合のデフォルト値

    3 番目にあげている他の項目とその個別の設定関数は次のものです。

    設定関数 対象
    set-file-name-coding-system ファイル名
    set-keyboard-coding-system キーボード
    set-terminal-coding-system ターミナル(コンソール)

    例えば、 Windows ではファイル名やキーボードは Shift-JIS(cp932) ですので、 prefer-coding-system で Shift-JIS 以外を指定した場合はこれらの変更は必須となります。
    デフォルトでは、キーボード入力はシステムのlocale設定にもとづいて変換されます。
    端末がlocaleにより暗に指定されるエンコードを実際にはサポートしない場合(たとえば、M-iをタイプしたときに非ASCII文字が挿入されるのに気づいたとき)、エンコーディングをオフにするためにkeyboard-coding-systemをnilにセットする必要があるでしょう。
    ターミナルの設定は Windows ではあまり使うことはないと思いますが、 GUI ではなく、ターミナルで Emacs を使う場合に必要となります。

    ここで注意点は prefer-coding-system では BOM 付きの UTF-8 (utf-8-with-signature) を指定してはいけないということです。
    サブプロセスでの文字コードも変わってしまうため、 grep やコンパイルなどで外部プログラムをちゃんと呼び出せなくなります。
    この対応としては default-process-coding-system 変数でプロセスの文字コードを変えればいいのですが、 それよりもデフォルトをシフトJISにしてファイルのデフォルトを変えた方が早いでしょう。

  4. ファイルのデフォルトの文字コード指定
    開いているバッファーのファイルの文字コードを変更する場合には set-buffer-file-coding-system (C-x RET f)を使います。
    しかし、これを設定で書いても意味がありません。
    バッファーごとの文字コード(buffer-file-coding-system) はバッファーローカルな値なので、 ロードしているバッファーの文字コードが変わるだけです。
    新規作成時のファイルのデフォルトを変える場合には set-default 関数を使って buffer-file-coding-system のデフォルト値を変更します。

    (set-default 'buffer-file-coding-system 'utf-8)

    以前はファイルのデフォルトの文字コード指定には default-buffer-file-coding-system 変数を使う方法もありました。
    しかし、 Ver. 23.2 から非推奨(obsolete)になったため、 buffer-file-coding-system を set-default で変更した方がよいでしょう。

  5. 文字コード基本設定まとめ

    ;;; 環境を日本語、UTF-8にする
;;
(set-locale-environment nil)
(set-language-environment "Japanese")
;;
;; デフォルトの文字コードを設定
;; prefer-coding-system の方は改行コードとのセットのシンボルも使える
(prefer-coding-system 'utf-8)
;; 同じような関数
;; (set-default-coding-systems 'utf-8)
;;
(set-file-name-coding-system 'utf-8)
(set-keyboard-coding-system 'utf-8)
(set-terminal-coding-system 'utf-8)
;; Windows の場合
;; (set-file-name-coding-system 'cp932)
;; (set-keyboard-coding-system 'cp932)
;; (set-terminal-coding-system 'cp932)
;;
;; ファイルのデフォルトの文字コード指定
(set-default 'buffer-file-coding-system 'utf-8)
;; バッファーローカルな値の変更にとどまるため設定で書いても無意味
;; (set-buffer-file-coding-system 'utf-8)
;; Ver. 23.2 から非推奨 (obsolete)
;; (setq default-buffer-file-coding-system 'utf-8)
;;
;; (setq locale-coding-system 'utf-8) ; システムのエラーメッセージや、format-time-stringのフォーマットやタイムスタンプなどの、システム文字列のエンコードおよびデコードで使用するコーディングシステムを指定

バイトコンパイル

  • エラーメッセージ
    • global/dynamic var `○' lacks a prefix
      Emacs Lispの慣例では、グローバル変数には packagename-variable 形式の名前を付ける必要があります。
      パッケージプレフィックスなしでグローバル変数に名前を付けることは、異なるパッケージ間で衝突を起こすことにつながります。

    • functions are not known to be defined
      declare-function をすればよい。

      (declare-function function "file" &optional arglist fileonly)
;; バイトコンパイラーにたいして、引数arglistをとるfunctionが定義されていて、その定義はfileにあるとみなすように告げます。

グローバル変数

  • 変数宣言
    defvar symbol [value [doc-string]]
    このスペシャルフォームは, symbolを変数として定義し, 初期値や説明文字列を設定する。
    この定義は, コードを読む人およびバイトコンパイラ向けに, 値を設定したり変更する変数としてsymbolを使うことを伝える。
    symbolの値が空でありvalueを指定してあると、defvarはvalueを評価してその結果をsymbolに設定し, symbolにすでに値があればvalueをまったく評価せず, symbolの値も変更しない（setq で変更する）。
    また、この定義には可能な限り、変数の意味と目的を説明するドキュメント文字列を含めるべき。名前の衝突を避けるように変数を命名することも行うべき。

    第2引数 value は常に設定しなくてもよく 、 defvar する際は通常 ファイルのトップレベル（関数宣言の前）で宣言する ことに注意。
    defvar を関数内で行ってしまうと、ファイル中の各関数でもいちいち defvar しなくてはならない。（自分には関数の実行順序が分かっても、バイトコンパイラにそれは判断できない。）
    そうするとバイトコンパイル時に "Warning: assignment to free variable `変数名" のようなエラーメッセージが出てくる。

変数/関数定義を削除する

変数/関数そのものを削除したい時に使える。

;; シンボル 'variable-hoge の変数を削除する
(makunbound 'variable-hoge)
;; シンボル 'function-hoge の関数定義を削除する
(fmakunbound 'function-hoge)

以下例

(setq foo 42)
>> 42
foo
>> 42
(makunbound 'foo)
>> FOO
foo
>> Error: The variable FOO is unbound.

また、 boundp はシンボル variable に対する現在の束縛が空でなければtを返す. さもなければnilを返す.

(boundp 'abracadabra)          ; 空で始める
     ⇒ nil
(let ((abracadabra 5))         ; ローカルに束縛する
  (boundp 'abracadabra))
     ⇒ t
(boundp 'abracadabra)          ; グローバルにはまだ空である
     ⇒ nil
(setq abracadabra 5)           ; グローバルに空でなくする
     ⇒ 5
(boundp 'abracadabra)
     ⇒ t

正規表現とバックスラッシュ

最重要概念！
Emacs Lisp では正規表現は「文字列」として表記する。
この引数として与えられる「文字列」は関数に利用される前に「評価」されるため、正規表現を記述する際は「評価した後の形」をイメージすればよい。

（通常の関数ではなく）正規表現エンジンを介する際の注意点
Emacs Lisp の正規表現エンジンに渡されたバックスラッシュは、メタ文字の意味を打ち消すメタ文字を意味する。
定義されてないメタ表現であれば、バックスラッシュの後の文字そのものを意味する。
例えば (re-search-forward "\\t") としても、正規表現には \t というメタ表現はないため、このバックスラッシュは無視され、't' だけに合致することになる。
\t を表現したければ タブを打ち消すための \ と、この \ を打ち消すための \ が必要となる。
すなわち、「正規表現として与える文字列」から「正規表現による解釈」までの間にバックスラッシュは「2度打ち消される」とイメージすれば良い。
（「関数に渡される前に行われる文字列の評価」と、「正規表現エンジンによって行われる評価」の二段階）

正規表現エンジンによる解釈 評価後の形（関数へ渡される形） 正規表現として与える文字列
タブ タブ \t
t \t \\t
\t \\t \\\\t
[]（文字列として） \ [ \ ] \\[\\]
.（文字列として） \. \\.
*（文字列として） \* \\*
\*（文字列として） \\\* \\\\\\*
\ 表現（正規表現における特別な意味） \文字 \\文字

文字列を評価する簡単な方法としては、*scratch* バッファに insert することが挙げられる。
eg. (insert "\\\\\\*") → \\\*
cf. (re-search-forward "\\\\\\*") → \* を検索

参考資料 : http://www.mew.org/~kazu/doc/elisp/regexp.html

エラーの発生の検知 (ignore-errors)

ingore-errors
エラーが返るとそのブロックからは nilが返る

(let ((result (ignore-errors
                (do-something)
                t)))
  (unless result
    (do-failed-case)))

do-somethingで errorが発生すると、エラーの発生を抑制し、 resultが nilになります。
成功するとそのままブロックの処理を続け tとなります。
あとはその結果に応じて失敗時の処理を行えば良いわけです。

変更内容による Emacs キーバインディングの設定方法

  • 既存のコマンドを引数を与えないで割り当てる -> そのまま

    (define-key global-map "\C-ca" 'fill-region)

  • 既成のコマンドに引数を与えて割り付ける -> 説明が必要 (ラムダ式で十分)

    (define-key global-map "\C-ca" 
  '(lambda ()
     (interactive)
     (backward-delete-char 2)))

  • 自作のコマンドを作成して割り付ける -> 説明が必要 (コマンドとは何かなど)

    (define-key global-map "\C-ca" 'my-command)
;;
(defun my-command ()
  (interactive)
  (insert "a"))

メジャーモードは一つのバッファに対して同時に一つしか設定できない。
一方マイナーモードはメジャーモードとも他のマイナーモードとも併用することができる。

アドバイス定義 (defadvice)

定義済みの関数に処理を付け足すことができる。

(defadvice 定義済み関数 (class 自分で決めた仮の関数名)
  処理)
;; 処理を付け足した関数を有効にする
(ad-activate '定義済み関数)

class では、 after か before か around を指定する。

包囲アドバイス

包囲アドバイスにより, 関数の元定義を包み込むLisp式を書けます.
このとき関数の元定義を実行する場所を特別なシンボルad-do-itで指定します.
包囲アドバイスの本体に現れたこのシンボルは, 元定義（と内側の包囲アドバイス本体）のフォームを含んだprognで 置き換えられます.

(defadvice foo (around foo-around)
  "Ignore case in `foo'."
  (let ((case-fold-search t))
    ad-do-it))

これは, fooの元定義を実行するときに 大文字小文字を区別しないで探索することを保証します.

ad-do-it は実際には変数ではないが, 変数のように用いる. 関数の元定義を実行する場所を指定する.
包囲アドバイスでad-do-itを用いなければ, 関数の元定義を実行しません. これは, 元定義を完全に無効にする手段です.

OSの違いを判定する

変数 system-type を用いる。
